Why it lands

Steam-like density, translated into a warmer operator surface.

The goal is not to mimic a game store literally. It is to borrow its strongest traits: immediate atmosphere, stacked discovery, and a sense that each module is worth opening.

Shelf logic

Feature clusters feel merchandised, not dumped.

Related capabilities are grouped like editorial collections: recipes, runtime adapters, progress, and safety controls all have clear frontage.

Warm palette

No black. Still high-contrast.

Instead of Steam's charcoal blues, the interface uses paper, copper, and amber light to keep the cinematic hierarchy without the coldness.

Operator confidence

Every action is explainable before it is clickable.

Install plans, progress events, secrets handling, and dashboard readiness are all surfaced as first-class product stories.

Feature shelf

A bento storefront built around operational moments.

This is the heavy-display zone: big modules, asymmetry, and tight captions, echoing storefront capsules without turning into a game listing clone.

01 · Progress events

Watch long operations move through named phases.

Instead of frozen buttons, operators get an evented sequence: preflight, plan generation, runtime install, readiness wait, and final success or failure.

02 · App center

One desktop shelf for AI apps, agents, and MCP projects.

Registry entries, recipe metadata, runtime adapters, and launch actions align into a single visual language.

  • Registry-backed cards
  • Recipe detail page
  • Status badges
  • My Library flow
03 · Secrets

Tokens are managed, not whispered.

Redaction and injection are treated as product features, not hidden implementation details.

04 · Ports

Resolve conflicts before they become support tickets.

Effective URL mapping and local collision handling are brought into the same narrative as install and launch.

05 · Rollback

Failure still has choreography.

Failed installs and uninstall flows are explicit recovery paths, not side effects the user must infer.

06 · Runtime router

The product value is in the routing discipline.

OpenNest is not rewriting every upstream app. It decides what environment each app needs, gets it ready, and opens the right dashboard with clean status and log surfaces.

FAQ

The usual objections, answered in product language.

Is this a clone of Steam for tools?

No. The reference is tonal and structural: dense discovery, editorial grouping, and strong shelf presence. The product itself remains a desktop control layer for OpenNest.

Why keep the page warm instead of dark?

Warm materials differentiate the product from generic devtool darkness while preserving contrast and cinematic depth.

Can this visual direction scale beyond a landing page?

Yes. The tokens, badges, cards, and section rhythm can extend into app center views, recipe detail pages, and status panels.

Ready for the front shelf

Ship the rollout with a page that feels curated, not assembled.

The visual system is already constrained: warm palette, L2 motion, asymmetric showcase panels, and clear operator language.
